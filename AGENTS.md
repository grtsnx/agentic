## Learned User Preferences

- Uses `bun` as the package manager, not npm; run scripts with `bun run <script>` (e.g. `bun run setup:pipeline`).
- Treats the plan/spec file `doc/instruct.md` as read-only — implement against it but do not edit it.

## Learned Workspace Facts

- NestJS provisioning pipeline lives under `src/pipeline/`; it creates a Vault, an Environment, and 26 Claude Managed Agents on the Anthropic beta Managed Agents API and writes their IDs to `output/`.
- Run the pipeline with `bun run setup:pipeline` (maps to `bun src/pipeline/pipeline.command.ts`).
- In `.env`, payment provider keys (Stripe, LemonSqueezy, Paddle, Paystack, PayPal), `HIGGSFIELD_API_KEY`, `UNSPLASH_ACCESS_KEY`, and `PEXELS_API_KEY` are optional; the pipeline skips/falls back gracefully when keys are missing.
- Media sourcing fallback chains: **Asset agent** images = Unsplash → Pexels (`PEXELS_API_KEY`) → keyless online (Openverse/Wikimedia/web_search) → generated on-brand SVG (never grey placeholders). **Video agent** = Higgsfield AI generation (`HIGGSFIELD_API_KEY`) → Pexels stock footage (`PEXELS_API_KEY`) → skip only if both missing. Pexels uses header `Authorization: $PEXELS_API_KEY` (raw, no "Bearer"); photos at `/v1/search`, videos at `/v1/videos/search`. Vault has a `Pexels API Key` env-var credential scoped to `api.pexels.com`, `images.pexels.com`, `*.pexels.com` CDNs + `player.vimeo.com`/`*.vimeo.com` (Pexels video files are Vimeo-hosted). Orchestrator gates video on `HIGGSFIELD_API_KEY || PEXELS_API_KEY`.
- Email "from" address domain is configured via the `EMAIL_DOMAIN` env var (the Email Agent interpolates it as `{custom}@${EMAIL_DOMAIN}`); do not hardcode a brand/domain name in code or README.
- The builder's brand name is configured via the optional `AI_NAME` env var; `src/main.ts` builds the Swagger/Scalar title as `${AI_NAME} — AI Builder` (defaults to `AI Builder` when unset). Never hardcode a brand name in the API docs/title.
- Reruns of `setup:pipeline` are idempotent (agents are matched by name and updated in place); `bun run archive:duplicates` cleans up duplicate agents from earlier runs.
- `bun run purge:all` (`src/pipeline/purge.command.ts`) wipes the Managed Agents workspace for a fresh start: deletes all sessions + vaults (permanent), archives all agents (the SDK has no agent hard-delete, only `agents.archive`), and clears local `output/agents.config.json`. Dry-run by default; pass `--yes` to execute and `--with-environments` to also delete environments. SDK methods: `sessions.delete`, `agents.archive`, `vaults.delete`, `environments.delete`; all `.list()` calls are async-iterable. Order: sessions → agents → vaults → environments. It also clears the auto-generated IDs from `.env` via `clearEnvVars` (`src/pipeline/lib/env-file.ts`): `ANTHROPIC_VAULT_ID` always, `ANTHROPIC_ENVIRONMENT_ID` with `--with-environments`, `ANTHROPIC_MEMORY_STORE_ID` with `--with-memory` — so no stale IDs remain after a purge.
- Refero MCP (`https://api.refero.design/mcp`, needs Refero Pro) is optional, gated on `REFERO_API_KEY`. When set, the vault gets a `static_bearer` credential bound to the MCP URL and the Design + Research agents attach the `refero` URL MCP server; when unset, both agents build without it. Config lives in `src/pipeline/config/mcps.config.ts` (`referoEnabled`, `REFERO_MCP_URL`).
- Component library is per-project: the Design Agent emits `DesignSpec.componentLibrary` (`shadcn` default for marketing sites, `heroui` for app/dashboard UIs). The CodeWriter branches on it.
- Managed agents can only use the built-in agent toolset + remote URL MCP servers. shadcn has no remote MCP (local stdio only) and HeroUI's remote Pro MCP needs a custom `x-heroui-personal-token` header the vault can't inject (vault only does `static_bearer` = `Authorization: Bearer`). So CodeWriter gets library knowledge via: distilled prompt rules, `web_search`/`web_fetch` for live docs (`TOOLS.CODE_WEB`), and the shadcn CLI over bash (`npx shadcn@latest add`) — not via shadcn/HeroUI MCP. Cursor-local MCPs/skills are not usable by the deployed pipeline agents.
- Uiverse (https://uiverse.io) is DIFFERENT from Animate UI / React Bits: it is COPY-PASTE only (MIT-licensed CSS/Tailwind elements) — NO shadcn registry / `/r/{name}.json` endpoint, NO CLI, NO MCP. So it must NOT be added to components.json `registries`. CodeWriter uses it as a SUPPLEMENTARY decorative source (loaders, glass cards, novelty CTAs — never form controls) by web_fetching uiverse.io, converting the snippet to TSX, and re-tokenizing colors→DesignSpec palette + spacing→allowed scale. Constant `UIVERSE_URL` in `src/pipeline/config/mcps.config.ts`. General rule learned: a library is only CLI/registry-installable if it exposes a shadcn `/r/{name}.json` endpoint; otherwise it's a web_fetch + re-tokenize source.
- Animated components come from two COMPULSORY shadcn-compatible registries: Animate UI (`https://animate-ui.com/r/{name}.json`, primary) and React Bits (`https://reactbits.dev/r/{name}.json`). Both are registries, NOT npm packages and NOT remote MCPs — the "shadcn MCP" (`npx shadcn@latest mcp`) is local stdio and can't attach to managed agents. Enforced at code-gen: the Animation Agent (`TOOLS.CODE_WEB`) installs components via the shadcn CLI (`npx shadcn@latest add @animate-ui/<c>` / `@react-bits/<c>`) and lists them in `AnimationManifest.animatedComponents` (`{name, registry, installCommand, importPath}`); CodeWriter imports those + installs extras, always generates `components.json` with BOTH `@animate-ui` and `@react-bits` registries mapped (for shadcn AND heroui projects), and adds `motion` as a dep. Agents web_fetch the registry sites to confirm exact names/peer deps. Constants in `src/pipeline/config/mcps.config.ts` (`ANIMATE_UI_REGISTRY`/`ANIMATE_UI_URL`, `REACT_BITS_REGISTRY`/`REACT_BITS_URL`).
- CI lives in `.github/workflows/ci.yml` (GitHub Actions, `oven-sh/setup-bun`, `bun install --frozen-lockfile`) with four parallel jobs: **audit** (`bun audit --audit-level=high`), **build** (`bun run build`), **test** (`bun run test:ci` = `jest --ci --passWithNoTests`), **smoke** (`bun run smoke`). The smoke target is `src/pipeline/smoke.ts` — it boots `PipelineModule` via `NestFactory.createApplicationContext` and resolves `PipelineService` to validate the whole DI graph (all 27 agents) WITHOUT calling `setup()`, so it needs no network or API key.
- Memory/persona uses Anthropic's MANAGED memory-store feature, not hand-rolled prompt injection. Research (docs: managed-agents/memory + /sessions) determined: a memory store (`memstore_…`) attached to a session is auto-mounted as a sandbox dir AND Anthropic auto-adds a note to each agent's system prompt pointing at it — agents read/write it with normal file tools (agent toolset required). Stores attach at SESSION creation via `resources[]` (`type:"memory_store"`, optional per-session `instructions` ≤4096 chars, `access` read_write|read_only) — NOT at agent creation. Since this repo only provisions agents/vault/env (not sessions), the correct change was to provision the store here and let the runtime app attach it. `MemoryStoreService` (`src/pipeline/memory/memory-store.service.ts`) getOrCreates `builder-memory`, seeds `persona.md` + `user-instructions.md`, and is non-fatal (returns '' + warns if the SDK lacks `client.beta.memoryStores` or the call fails). ID persisted to `.env` as `ANTHROPIC_MEMORY_STORE_ID` and included in `agents.config.json`. Did NOT build a prompt-injecting MemoryService (would fight the managed feature). `purge:all` deletes memory stores only with `--with-memory` (SDK: `memoryStores.list/delete`, `memoryStores.memories.create`).
- Vault + environment are auto-provisioned and written back to `.env`: `vault.service.ts` and `environment.service.ts` already `getOrCreate` (create when `ANTHROPIC_VAULT_ID`/`ANTHROPIC_ENVIRONMENT_ID` are blank, else reuse). After both resolve, `pipeline.service.ts` calls `upsertEnvVars` (`src/pipeline/lib/env-file.ts`) to persist both IDs into `.env` (in-place rewrite preserving other lines + inline comments; creates the file if missing), so the first run creates them once and every later run reuses them. Only `ANTHROPIC_API_KEY` is truly required now; the two IDs are "auto-created (leave blank)" in `.env.example`/README.
- User-uploaded references are honored end-to-end via `IntentSpec.mediaSignal[]` (images/logos/brand kits/URLs) + `designSignals.referenceUrls`. The **Design Agent** uses `TOOLS.READ` (read = vision, so it can directly open uploaded images and pull real hex/typography, not just URLs) and its prompt mandates user references override defaults. The **Asset Agent** (`TOOLS.BASH_READ`) ingests mediaSignal assets FIRST (curl URL or read attachment → upload to R2) into proper slots (`logo`/`hero`/`gallery`) with `source:"user"`, and only fills remaining slots with Unsplash — user assets are never overwritten. The Orchestrator data-flow section explicitly routes mediaSignal/referenceUrls to design + asset.
- Besides the provisioning pipeline (`src/pipeline/`), there's a separate **runtime HTTP API** (`src/runtime/`, bootstrapped by `src/main.ts` via `NestFactory.create(AppModule)`) that an external frontend (e.g. Next.js) calls to drive builds: `POST /builds` (start → `{buildId, sessionId}`), `GET /builds/:id` (status), `GET /builds/:id/stream` (live orchestrator events as SSE / `text/event-stream`, frames are JSON `{kind, payload}` — yes, build progress streams in real time), `POST /builds/:id/approve` (resume past the preview gate to deploy). OpenAPI docs are served two ways: Swagger UI at `/docs` (raw JSON `/docs-json`) and Scalar at `/reference` (`@scalar/nestjs-api-reference`, theme `purple`); the Swagger title is `${AI_NAME} — AI Builder` and the description is just `AI Builder` (no long sentence). Start the runtime API with `bun run start` (or `bun run start:dev` = `nest start --watch`); on boot `main.ts` logs the runtime URL plus the Swagger UI (`/docs`), Scalar (`/reference`), and OpenAPI JSON (`/docs-json`) URLs to the terminal. CORS via `CORS_ORIGIN` env (comma-split, defaults `*`); port via `PORT` (default 3000).
- The frontend UI for this builder is a **separate Next.js repo `mayaweb`** at `/Users/grtsnx/programming/agent/mayaweb` (Next.js 16, React 19, Tailwind v4, shadcn radix-maia style, hugeicons, zustand, next-themes; package manager `bun`). It was rebuilt Lovable-style (left chat panel + right workspace with Preview/Code/Activity tabs) and is wired to the REAL runtime API (no longer mocked). It talks to the runtime via same-origin Next proxy route handlers under `app/api/builds/*` that forward to a configurable `RUNTIME_URL` env (default `http://localhost:3000`); the SSE/stream route pipes the upstream `text/event-stream` body through verbatim (keeps the backend URL server-side, avoids CORS). Frontend brand name is configurable via `NEXT_PUBLIC_APP_NAME`. Both the runtime API and `mayaweb` default to port 3000, so run the frontend on another port in dev: `PORT=3001 bun run dev`.
- mayaweb frontend gotchas: hugeicons `@hugeicons/core-free-icons` export names must be verified before importing — the code/terminal icons are `CodeIcon`/`TerminalIcon`, NOT `Code01Icon`/`Terminal01Icon` (grep of the minified bundle is misleading; trust the TS ESM exports); the icon-prop type is `IconSvgElement` from `@hugeicons/react`. The new react-hooks lint rule flags synchronous `setState` in an effect body (e.g. the next-themes "mounted" hydration pattern) — defer it (e.g. `setTimeout(..., 0)`) to stay lint-clean.
- There are now 27 agents (was 26). A dedicated **Security Agent** (`src/pipeline/agents/security.agent.ts`, `AGENT_MODELS.security`) runs in the parallel quality gate (with QAS/Accessibility/Performance) and does dependency CVE scanning (`npm audit`/osv-scanner in Daytona) + SAST (injection, XSS, SSRF, secret exposure, broken authz/InsForge RLS). Critical findings route to the **AutoFix Agent**, which now has two modes: build-error recovery (RunDev) and scan-finding remediation (Security/QAS). After AutoFix patches, the scanner re-scans; criticals unfixable after 3 attempts are surfaced to the user and BLOCK deploy. When editing agent counts, update orchestrator (`description`, "manage N", metadata note), `pipeline.service.ts` log, and README (`The 27 agents` heading + anchors).
